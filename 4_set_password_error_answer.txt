
TCG Opal 3.2.5.4 User1 Password Change Error Analysis

작성해주신 '4. set_password_error.txt' 파일의 내용을 꼼꼼히 확인했습니다.

현재 상태를 요약해보면, Admin1으로 세션을 열고, Admin1의 비밀번호를 설정하고, User1 권한을 활성화하는 것(3.2.5.2 및 3.2.5.3)까지는 결과 상태(Status)가 0(Success)으로 정상 작동했습니다. 그런데 3.2.5.4 단계에서 User1의 비밀번호를 변경하려고 할 때, 응답(Response)이 완전히 오지 않고 중간에 끊기는 현상이 발생하고 있습니다.

이러한 "회신이 일부 오다가 중단되는 현상"은 TCG Opal 통신에서 권한 부족(Not Authorized) 같은 논리적인 오류가 아니라, 드라이브의 TCG 파서(Parser)가 패킷을 해석하다가 충돌(Crash)하거나 프로토콜 스트림 오류로 인해 연결을 강제로 끊어버렸을 때(Abort) 나타나는 가장 전형적인 증상입니다.

성공적으로 진행된 앞 단계의 코드를 복사해서 수정하시면서 TCG 패킷의 바이트 인코딩 규칙이나 길이(Length)가 어긋났을 확률이 매우 높습니다. 

다음의 4가지 주요 원인과 해결 방법을 확인해 보시기 바랍니다.

## 문제 해결을 위한 체크리스트

### 1. 패킷 길이(Length Header) 값의 오류 (가장 유력함)
앞 단계인 3.2.5.2(Admin1 비밀번호 변경) 코드를 재사용하면서 <User1_password>를 넣을 때, 비밀번호 문자열의 길이는 달라졌는데 패킷 헤더(ComPacket Header 또는 DataStream Length)에 기록되는 전체 길이 값은 업데이트하지 않았을 가능성이 높습니다.
- TCG 토큰 스트림은 바이트 단위의 길이가 정확해야 합니다. 토큰의 바이트 배열 길이가 실제 데이터 길이와 다르거나 패킷 전체 길이와 불일치하면, 드라이브가 나머지 데이터를 기다리다 타임아웃 되거나 버퍼 오버플로우로 응답 스트림이 끊어집니다.
- 해결 방안: User1 비밀번호 데이터 길이를 동적으로 계산하여 패킷 길이를 업데이트하도록 코드가 작성되었는지, 하드코딩된 길이 값이 들어가 있지 않은지 확인하세요.

### 2. Start (0xF0, 0xF2) / End (0xF1, 0xF3) 토큰 쌍의 불일치
Set 메소드의 파라미터 구조를 조작하다가 EndName (0xF3)이나 EndList (0xF1) 태그를 누락했거나 잘못 삽입했을 수 있습니다.
- 3.2.5.4 단계의 Values 파라미터는 다음과 같은 토큰 구조를 정확히 닫아주어야 합니다.
  StartName (0xF2) 
    "Values" (1) 
    StartList (0xF0) 
      StartName (0xF2)
        "PIN"(Column 3)
        <User1_password_Bytes>
      EndName (0xF3)
    EndList (0xF1)
  EndName (0xF3)
- 해결 방안: 바이트 배열을 덤프(Dump)하여 페이로드 내에 Start와 End 쌍이 제대로 매칭되었는지 확인하세요.

### 3. 대상 객체(Invoking UID) 값이 잘못 설정되었는가?
3.2.5.4 단계에서 비밀번호를 설정해야 할 대상은 C_PIN_Admin1이 아니라 C_PIN_User1 입니다.
- C_PIN_Admin1의 UID: 00 00 00 09 00 03 00 01
- C_PIN_User1의 UID: 00 00 00 09 00 03 00 03
- 해결 방안: Request 패킷을 구성할 때 Invoking UID 항목에 00...03 이 정확히 들어가 있는지 확인하세요. 만약 존재하지 않는 객체나 잘못된 형식으로 접근하면 파싱 에러로 인해 응답이 끊길 수 있습니다.

### 4. 동기화/타임아웃(Sync / Timeout)
만약 길이가 맞는데도 끊긴다면 패킷 하단의 4바이트 정렬(Padding) 문제일 수도 있습니다. 앞 단계들의 토큰 페이로드는 4의 배수로 딱 맞아 떨어졌는데, User1 password는 길이가 달라서 0x00 크기 패딩(Padding) 처리가 누락되었을 수도 있습니다.

---

요약하자면:
권한 문제가 있었다면 드라이브를 정상적으로 파싱하고 상태를 0x01 (NOT_AUTHORIZED)로 회신했을 것입니다. "응답이 가다가 끊긴다"는 것은 100% 드라이브 측의 TCG 패킷 디코딩(Parsing) 실패를 의미합니다. <User1_password>를 삽입하는 부분의 "토큰 길이 헤더(Length)"와 "패딩(Padding)" 처리를 집중적으로 디버깅해 보시기 바랍니다!
